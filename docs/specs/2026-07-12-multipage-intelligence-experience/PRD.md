# PRD：多页 AI 行业决策情报产品

## 1. 当前问题

1. 品牌宣言占据首页首屏，用户进入后不能立即消费“今天发生什么、为何重要”。
2. 六条主线、时间轴和四个工具被压在同一长页中，信息密度和可分享性都不足。
3. 所有内容共用一个 title；事件和工具缺少稳定 URL，浏览器返回、分享和搜索引擎理解都较弱。
4. `app.js` 首次加载七份 JSON；未打开工具也加载其数据。
5. 42 项能力、10 项评测、33 个角色和完整 Changelog 大量被切片隐藏。
6. GitHub Stars 没有进入产品反馈环；README 默认中文，不利于国际传播。
7. Issues 不能提交、验证或维护来源；来源审计没有跨 Actions 累积检查。
8. snapshot 中第三方摘要过长，缺少公开版权、纠错和下架路径。

## 2. 产品目标

- 首页第一屏直接展示一件具备一手证据的今日判断，包含事实、重要性、影响对象、下一观察与证据动作。
- 建立稳定多页：主页、主线总览、六条主线详情、时间轴、事件详情、四个工具、Changelog、来源、版权。
- 每个页面拥有合理 title、description、canonical、导航 active 状态、loading/empty/error/stale 状态。
- 首页消费其他页面的摘要，而不是复制完整内容；品牌宣言移动到 Footer 前。
- Changelog 以版本轨道、能力增量和指标变化呈现，避免新闻 ticker。
- GitHub Stars 在构建阶段获取并写入静态 DTO，超过 24 小时则退化为无数字 CTA。
- 默认英文 `README.md`，提供完整 `README-zh-cn.md`。
- Issues 来源 proposal 经过不可信输入验证、维护者审批、draft PR、CI、shadow 和 E3/E4 门禁。
- Actions audit/refresh 使用同一写入锁和仓库快照，让健康检查可以跨运行积累。
- 代码 MIT 与第三方内容权利明确分开；公开仓库只保留必要 metadata、短摘录和 Agent Pulse 原创分析。

## 3. 非目标

- 不引入依赖 404 fallback 的 React SPA。
- 不允许公开 Issue 直接修改 main、选择内部 adapter、设置分数或激活来源。
- 不在浏览器运行时依赖 GitHub API、远程 iconfont 或网络 SVG。
- 不重写已有 Git 历史；历史版权风险另行评估。
- 不把页面拆分等同于内容质量提升；证据门禁仍由数据层负责。

## 4. 信息架构

```text
/
├── lines/
│   └── {track}/
├── timeline/
├── events/{slug}/
├── scout/
├── actors/
├── resources/
├── product/
├── changelog/
├── sources/
└── legal/
```

首页顺序：

1. Today：直接展示事实与判断。
2. 六条主线摘要：当前 thesis、节点数、最新转折、详情入口。
3. 最近证据：5 个含一手证据的时间节点。
4. 决策工具入口：星探、角色、资源、系统水位的真实数量与摘要。
5. 最新 Changelog：最近 2 个版本的用户价值与能力变化。
6. GitHub CTA：Stars、版本、更新时间。
7. 品牌宣言与版权 Footer。

## 5. 页面完成标准

- 正文桌面不低于 14px，关键正文不低于 16px；主要触控目标至少 44×44px。
- 1440×900 首屏装饰空白不超过 25%；主页首屏完整看见 Today 事实与至少两个判断维度。
- 390×844 无全局横向滚动；移动端提供固定底部导航或等价可发现导航。
- 从摘要到原始证据不超过两次交互。
- 每个独立页首屏至少包含一个核心结论、一个数据状态和一个下一动作。
- 主线详情至少展示当前判断、下一信号、阶段轨迹、5 个证据节点、中国/全球位置、四角色视角与证据缺口。
- 时间轴桌面首屏至少看见 5 条摘要与一个详情；筛选和选中状态可复制、刷新和返回。
- 工具页不得通过 `slice()` 隐藏主体数据；资源页必须展示价格/单位/核验时间，产品页必须展示完整评分惩罚。
- 最新 Changelog 默认展开，至少五个历史版本可访问。
- 每页独立 title；事件页 title 使用事件标题。

## 6. 高质量水位

最终验收不以“页面数量”判断，而以四个问题判断：

1. 用户是否在第一屏获得可以复述的判断？
2. 用户是否能沿主线和时间找到证据如何累积？
3. 用户是否能理解系统不知道什么、为什么分数不高？
4. 用户是否能安全地贡献来源、纠错和追踪产品演进？
