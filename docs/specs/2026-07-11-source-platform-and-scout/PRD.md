# 数据源平台与星探精灵 PRD

- 状态：Stage 2 foundation 已实现；本文同时定义后续完整 Source Platform 与 Scout 验收边界
- 日期：2026-07-11
- 目标阶段：Stage 2（可信数据底座）至 Stage 3（行业认知引擎）
- 前置规格：`docs/specs/2026-07-11-agent-pulse-rebuild/`

## 1. 背景与水位

当前项目已经具备 Node.js/TypeScript、SQLite、统一 adapter、来源 lifecycle/health、SourceRun、有界并发、瞬时错误重试、条件请求、事件/主线模型、本地管理台、Scout v1 和静态页面骨架，但还不能称为成熟的行业情报系统：

- 已注册来源数量和国内一手来源覆盖不足；
- AI HOT fingerprint、分页 cursor、来源级 scheduler/per-host budget 和完整 state 事务尚未形成闭环；
- 已有基础降级/隔离/恢复/软退役，但缺少基于长期 SLO 的自动晋级、完整熔断和 package 卸载验证；
- 热点缺少真实的跨平台、跨地区传播证据；
- 自动事件收敛仍会产生待编辑占位文本；
- 管理台修改、本地静态快照和 GitHub Pages 尚未形成可审计发布闭环；
- 星探已有证据型 v1，但个性化 owner fit、效果评估和长期反馈校准仍未成立。

因此本规格不把现有能力标记为“已完成”，而是定义下一阶段可验收的建设边界。

## 2. 产品目标

### 2.1 数据源平台

把来源从一条可启停配置升级为可发现、可验证、可运行、可降级、可维护、可卸载的一等资产，使系统能够长期稳定地观测国内外 AI 产业变化。

平台应回答：

1. 我们正在观察谁，为什么值得观察？
2. 哪些重要角色、地区、传播域和主题仍存在覆盖缺口？
3. 每个来源如何获取、是否合规、多久更新、当前是否健康？
4. 一条信号来自哪里，是否回到了原始事实源？
5. 某个来源失败或结构漂移时，系统如何降级并恢复？

### 2.2 星探精灵

把已经审核的行业事件和主线变化进一步转化为面向系统主人的机会卡，帮助其提升认知、沉淀产物、扩大影响力并提升个人价值。

星探不是“随机灵感生成器”，也不是事实发布器。它要基于证据，输出可验证的小行动，并通过主人反馈持续学习偏好。

## 3. 非目标

- 不绕过登录、付费墙、WAF、验证码或平台访问限制。
- 不镜像第三方完整文章、聚合站生产数据库或受限价格数据。
- 不在 Stage 2 承诺覆盖所有国内外账号；先建立覆盖地图、优先级和可持续接入能力。
- 不让 LLM 自动发布事实、自动执行投资/交易或未经确认的外部动作。
- 不在缺少真实传播数据时，把人工分数描述成规模化社交热点。
- 不因“自我生长”而允许运行时静默修改 schema、评分或发布策略。

## 4. 用户与核心任务

### CEO / 业务负责人

- 看清能力、分发、成本、政策和竞争结构的关键变化；
- 得到“现在应该验证什么”的行动建议；
- 区分真正结构性变化与单平台噪声。

### 投资负责人

- 沿技术、资本、商业化和中国追赶主线追踪角色进展；
- 查看证据质量、独立验证和反例；
- 获得可转化为 thesis、访谈提纲或尽调清单的星探卡。

### 系统主人 / 编辑

- 查看来源覆盖、健康和失败原因；
- 发现新来源，试拉取并按质量证据晋级；
- 审核信号、聚类、事实、洞察、主线和星探建议；
- 发布可回滚的静态快照。

## 5. 数据源产品模型

### 5.1 Source

来源定义至少包含：

- 身份：名称、slug、owner、主页、来源类型、人物/机构关系；
- 分类：tier、role、地区、语言、主题、受众、传播域；
- 获取：adapter、endpoint、frequency、quota、timeout、cache/cursor 能力；
- 治理：robots、license、attribution、content policy、成本和维护人；
- 质量：authority、freshness、originality、stability、coverage、independence；
- 运行：lifecycle、health、last success、failure streak、next run、adapter version。

### 5.2 Source Package

每个来源包应包含：

- manifest 与 capability 声明；
- 配置和输出 schema；
- adapter 实现；
- fixture/golden samples；
- contract、漂移、错误和降级测试；
- 合规说明、速率限制和归属；
- 版本迁移、停用和卸载策略。

### 5.3 生命周期

```text
discovered -> draft -> shadow -> active
                         |         |
                         v         v
                    rejected   degraded -> quarantined
                                      \-> retired
```

- `discovered`：只记录候选，不执行生产采集；
- `draft`：完成合规和接入设计；
- `shadow`：低频、小窗口、隔离验证；结果不自动进入发布链路；
- `active`：达到晋级门槛，进入正常调度；
- `degraded`：仍可提供部分数据，但新鲜度或完整性下降；
- `quarantined`：结构漂移、合规或安全风险，停止进入事实链；
- `retired`：软卸载，不再调度，但历史证据和运行记录仍可读取。

## 6. 来源发现与晋级流程

### 6.1 候选发现

候选可以来自：

- 现有事件回链中反复出现的原始域名；
- 关键 Actor 的官网、博客、论文页、GitHub 和监管披露；
- 聚合站暴露的原始链接；
- 专家、记者、CXO、投资机构和社区白名单；
- 主线/地区/语言覆盖缺口扫描；
- 管理员手工提名。

### 6.2 候选评分

候选分与事件热度分分离，至少考虑：

- 原创性和一手事实比例；
- 历史准确性与更正透明度；
- 新鲜度和发布稳定性；
- 对既有覆盖的增量价值；
- 身份和媒体矩阵独立性；
- 获取方式稳定性、合规性和成本；
- 对中国/海外、技术/资本/商业化等覆盖缺口的贡献。

### 6.3 晋级门槛

`draft -> shadow`：

- 归属、robots/license、获取方式和速率限制已记录；
- manifest、schema、adapter、fixture 和 contract test 齐备；
- 不需要绕过访问限制；
- URL 与外部文本安全边界通过测试。

`shadow -> active`：

- 连续观测窗口内成功率、新鲜度和重复率达到配置门槛；
- 至少一次 schema drift/错误路径演练通过；
- 能保留原始来源 URL 和采集 provenance；
- 数据样本经人工抽检，来源角色和 tier 合理；
- 未把聚合站内容误当成事实终点。

晋级、降级和恢复都必须记录操作者、理由、证据和时间。

## 7. 聚合站与事实回源

- AI HOT、HuggingNews 等聚合站只承担候选发现、事件聚类提示和传播热度补充。
- 聚合条目必须尽可能提取并保存原始 URL；事实确认必须回到公司公告、论文、监管文件、官方 release 或独立专业验证。
- 如果聚合站没有提供可访问的原始事实源，该条目只能保持 discovery/review 状态。
- 同一聚合站的多个条目不能被计为多个独立事实来源。
- 聚合摘要可以帮助检索，但不能直接复制为公开事实陈述。

## 8. 星探精灵产品设计

### 8.1 输入

- 已审核 Event 及 evidence；
- Track 最近阶段、转折和覆盖缺口；
- Actor 进展、国内外对标和资源成本变化；
- 主人的关注方向、能力、可投入时间和历史反馈；
- 已有产物、选题和机会，避免重复。

### 8.2 输出类型

- `venture`：创业方向或可验证需求；
- `media`：文章、视频、演讲、播客或数据栏目选题；
- `work`：产品、工程、研究或组织动作；
- `learning`：需要补齐的认知、方法和关键问题；
- `artifact`：可沉淀的数据集、框架、报告、工具或开源项目；
- `influence`：有证据支撑的观点表达和影响力动作。

### 8.3 Opportunity Card

每张机会卡至少包含：

- 标题与类型；
- 触发事件、主线和证据链接；
- 目标用户/受众；
- 为什么是现在；
- 非共识或信息差；
- 建议产物；
- 1–3 天可执行的首个小实验；
- 预期价值、风险与失效条件；
- novelty、actionability、owner fit、timing、evidence strength 分数；
- 状态和主人反馈。

### 8.4 交互

- 星探以克制的“精灵冒泡”方式出现，但不得打断核心阅读；
- 支持收藏、展开、转为任务/选题、暂缓、驳回和说明原因；
- 同一事件默认限制机会数量，避免泛化生成；
- 可以按创业、自媒体、工作和认知视角筛选；
- 管理台可调整频率、阈值、偏好、禁区和公开边界。

### 8.5 治理

- 星探卡是建议，不是事实；必须显式区分触发证据与模型推断。
- 不自动发布、不自动联系外部人员、不自动交易或创建公开承诺。
- 缺少证据或只由单一聚合信号触发时，不生成高置信机会卡。
- 记录 prompt/model/version、输入 evidence IDs 和人工反馈，支持回放和评估。

## 9. 验收标准

### 数据源平台

- 管理台可完成来源发现、试拉取、晋级、降级、隔离、恢复和软卸载。
- 每个 active 来源均有 manifest、fixture、contract test、合规记录和 owner。
- ETag/Last-Modified/cursor/fingerprint 能按来源原子保存；失败不会推进状态。
- 单来源超时、429、5xx、解析异常或空结果不会中断整批任务。
- 连续失败触发降级/熔断，成功 probe 后才恢复正常调度。
- 每个聚合条目都能显示回源状态；无原始事实源的内容不能发布为事实。
- 来源覆盖面板能显示国内/海外、tier、角色、主题和传播域缺口。

### 星探

- 机会卡必须绑定至少一个已审核事件和可点击证据。
- 六种输出类型可筛选，支持收藏、暂缓、驳回和反馈。
- 同一事件的重复机会会被聚类或抑制。
- 机会卡完整包含 why now、首个实验、风险和失效条件。
- 低证据输入不会被标记为高置信建议。
- 星探结果默认私有，不进入静态公开站，除非管理员明确审核发布。

### 诚实发布

- README、状态页和 TASKS 区分 implemented、experimental、planned。
- 手工 seed 和演示分数明确标记，不能作为真实热点运行证据。
- MySQL、UI smoke、跨平台热度等能力只有在对应测试通过后才宣称支持。
