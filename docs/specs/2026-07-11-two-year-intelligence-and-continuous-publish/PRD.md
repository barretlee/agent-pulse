# 两年行业情报回溯与持续发布 PRD

- 状态：Proposed
- 日期：2026-07-11
- 历史窗口：2024-07-01 至 2026-07-11（含首日；末日以实际 snapshot `asOf` 为截止点；时间入库统一 UTC）
- 当前水位：Stage 1.2 / Stage 2 foundation；历史回溯、阶段总结和持续发布闭环尚未实现
- 关联规格：`docs/specs/2026-07-11-source-platform-and-scout/`

## 1. 背景

当前公开站以少量人工 seed 事件展示产品骨架，无法支撑“沿时间轴理解行业如何走到今天”。只看最近消息还会产生三类认知缺口：

1. 看得到发布，看不到技术、资本、产品与商业化之间的传导；
2. 看得到海外起点，看不到中国厂商从跟随、并跑到局部领先的时间差；
3. 看得到今天的结论，看不到当时证据、分歧与判断如何演变。

本项目需要建立 2024-07 至 2026-07 的两年基线，并让历史回填与实时更新长期并行。目标不是批量补新闻，而是构建可回源、可去重、可分阶段阅读、可持续发布的行业认知时间轴。

## 2. 产品目标

- 回溯两年内真正改变 AI 能力、产业、产品、资本和应用格局的关键事件；
- 为每个公开事实保留原始证据、时间精度和历史语境；
- 将同一事件的多篇报道、多个语言版本和多平台观察收敛为唯一 Event；
- 沿技术、AGI、投资、商业化、产品、To C/B/D/G、中国追赶主线组织历史；
- 产出月度脉冲、季度阶段总结、年度结构变化与两年总览；
- 历史回填不阻塞当天情报，二者通过同一质量门禁进入可回滚静态快照；
- 明确区分“当时可知事实”“后来验证结果”和“今天的回看判断”，降低后见之明偏差。

## 3. 非目标

- 不追求收录两年内所有 AI 新闻；数量不是完成指标。
- 不重建或镜像第三方全文、社交平台完整历史或受限数据库。
- 不因历史社交指标缺失而伪造热度；不可比的数据标记为未知。
- 不用今天的结果倒推当时一定可知的结论。
- 不绕过登录、WAF、验证码、付费墙或平台访问限制。
- 不在缺少真实 MySQL、浏览器和发布闭环验收时宣称生产完成。

## 4. 目标用户与核心问题

### CEO / 业务负责人

- 哪些能力变化已经跨过产品化阈值？
- 分发、成本和组织能力如何改变竞争结构？
- 哪些变化是周期噪声，哪些是不可逆转折？

### 投资负责人

- 资本在模型、算力、应用和基础设施之间如何迁移？
- 哪些融资/并购对应真实能力或商业验证，哪些只对应叙事？
- 中国角色与全球基准的差距在哪些季度收窄或扩大？

### 技术 / 产品负责人

- 技术路线如何从论文、开源进入 API、产品和企业采购？
- 能力、可靠性、延迟、成本和可控性分别何时发生变化？
- 面向消费者、企业、开发者和政府的成熟节奏有何不同？

### 系统主人 / 编辑

- 哪些月份、角色、地区和主线仍存在证据缺口？
- 一条历史事件为何发布、为何归入某个阶段、是否需要修订？
- 如何在不影响当天情报的情况下安全回填并持续发布？

## 5. 主线定义

一个 Event 可以进入多条主线，但事实只保存一份。每条关联保存该主线下的阶段、节点角色、叙事和顺序。

### 5.1 技术演进

观察模型架构、训练/推理、Scaling、多模态、Agent、代码、搜索、世界模型、具身智能、AI Infra、芯片和开源工具链。

关键判断：能力边界、单位成本、可靠性和工程可用性是否发生结构变化。

### 5.2 AGI 进展

观察泛化、自主规划、长期任务、工具使用、记忆、环境交互、可靠性、安全和治理。

关键判断：进展是 benchmark 提升、受控演示，还是跨任务、跨环境、可复现的能力跃迁。

### 5.3 投资

观察融资、估值、并购、战略投资、CapEx、算力资产、人才流动和产业链资本传导。

关键判断：资本在验证什么假设，风险和收益向哪个层级迁移。

### 5.4 商业化

观察收入、定价、推理成本、毛利、留存、席位渗透、分发、采购、渠道和商业模式。

关键判断：能力是否转化为持续付费、可交付 ROI 和可扩张单位经济。

### 5.5 产品

观察模型能力如何进入助手、搜索、办公、创作、编码、客服、教育、硬件、机器人和行业应用。

关键判断：产品是否改变用户行为、工作流入口、交互范式或平台控制点。

### 5.6 To C

观察个人订阅、免费层、用户增长、留存、创作者工具、陪伴、搜索和终端入口。

### 5.7 To B

观察企业席位、私有数据、权限、审计、工作流、行业交付、安全、采购和 ROI。

### 5.8 To D

观察模型 API、SDK、Agent runtime、开发工具、开源生态、兼容协议、价格和开发者分发。

### 5.9 To G

观察政策、监管、政府采购、公共服务、主权 AI、国家算力、安全评测和产业扶持。

### 5.10 中国追赶

在同一能力、产品、价格、开源、商业和供应链节点上，对比中国与全球基准，阶段统一为：

- `following`：主要复现海外已验证方向；
- `closing`：差距有可验证收窄；
- `parallel`：同一窗口形成可比能力/产品；
- `leading`：在明确维度形成可复现领先；
- `constrained`：因算力、供应链、政策或生态受限；
- `globalizing`：产品、开源或商业进入海外市场。

不得用公司整体标签代替节点证据；同一角色可在不同维度处于不同阶段。

## 6. 历史 Event 标准

### 6.1 必填内容

- 一句话事实；
- 发生时间、时间精度和首次公开时间；
- 事件背景与当时行业状态；
- 核心变化及关键数字；
- 技术、产业、产品或资本含义；
- 当时可知判断与不确定性；
- 后来验证/反证，仅在明确标注的回看字段呈现；
- 业务价值和后续观察；
- 关键词、Actor、主线、阶段；
- 原始证据和独立验证。

### 6.2 事实发布门槛

满足以下任一条件：

1. 至少一个可访问、身份可确认的 Tier 1 原始来源；
2. 原始来源已消失但存在可信存档，且至少一个独立 Tier 2 验证；
3. 两个相互独立的 Tier 2 来源，事件明确标记为“缺少原始档案”。

监管、融资金额、财务数字、安全事故和重大能力声明优先要求原始文件。团队自测、公司口径和独立复现必须分开。

### 6.3 不发布条件

- 只有聚合摘要、搜索片段或无法确认身份的转述；
- 时间、主体或核心动作存在未解决冲突；
- 只是同一公告的机械转载，没有新增独立验证；
- 只有今天的概括，无法恢复当时证据；
- 仍含“待编辑”占位、无证据洞察或无法解释评分。

## 7. 时间与历史语境

每条事件至少区分：

- `occurredAt`：实际发生时间；
- `publishedAt`：来源首次公开时间；
- `observedAt`：Agent Pulse 获取时间；
- `verifiedAt`：编辑最后核验时间；
- `asOf`：某段判断成立时所使用的信息截止时间；
- `timePrecision`：minute、day、month、range 或 unknown。

历史详情分成两个区域：

1. **当时视角**：只使用 `asOf` 之前可知信息；
2. **回看验证**：明确写出后来发生的验证、反证和修订。

## 8. 热度、影响与重要性

历史事件分别记录：

- `heatStatus=observed`：有当时真实、可解释的传播观测；
- `heatStatus=reconstructed`：由多个存档/独立响应谨慎重建；
- `heatStatus=unknown`：缺少可比数据，不给虚假数值。

影响和历史重要性不能只由传播决定。入选两年主线至少满足一项：

- 改变能力/成本/可靠性边界；
- 形成产品或平台控制点；
- 改变资本、竞争、供应链或监管结构；
- 获得持续商业验证；
- 成为后续多事件共同依赖的转折节点；
- 为中国追赶提供可比较的基准或反例。

## 9. 去重与事件边界

- canonical URL 与内容 hash 做文档级去重；
- 同一文档被多个来源观察时保留多个 Observation，不重复创建事实文档；
- 标题、实体、claim、时间窗口和原始引用链共同决定 Event 候选；
- 多语言翻译和转载归入同一事件，保留各自传播观察；
- 产品首次发布、重大版本、价格变化和商业扩张原则上拆为不同 Event；
- 同日多项公告只有在核心 claim 和业务动作一致时合并；
- 冲突版本不强行合并，进入 contradiction/review 队列；
- 自动合并和人工拆分都保留决策理由与审计。

## 10. 阶段总结

### 10.1 层级

- 月度 Pulse：本月新事实、升温主题和覆盖缺口；
- 季度 Phase：阶段问题、关键节点、转折、主导/竞争路线；
- 年度 Arc：能力、资本、产品与商业化的结构变化；
- 两年 Synthesis：2024-07 到 2026-07 的主线连接、中国相对位置和未来观察。

### 10.2 每条主线总结模板

- 这一阶段的核心问题；
- 起点状态与结束状态；
- 3–7 个关键证据节点；
- 形成转折的驱动力；
- 主导路线与竞争假设；
- 国内外时间差和角色变化；
- 对 CEO、投资、技术和产品的意义；
- 尚未解决的矛盾与下一阶段观察；
- 证据覆盖和已知缺口。

总结必须引用已审核 Event，不得直接从原始 feed 生成；阶段结论也要支持修订和版本记录。

## 11. 历史回填批次

按时间而不是按来源一次性灌库：

1. 2024-07 至 2024-12；
2. 2025-01 至 2025-06；
3. 2025-07 至 2025-12；
4. 2026-01 至 2026-06；
5. 2026-07-01 至 2026-07-11。

每批先建立 source/archive coverage，再采集 Document/Observation，最后聚类、审核、编排和发布。上一批发现的实体别名、转载关系和质量问题必须反馈到下一批。

## 12. 持续发布体验

- 实时 lane 处理当天/近期来源，优先级高于 backfill lane；
- 历史候选按月分区、限额运行，可暂停和恢复；
- 新历史事件进入 preview，不直接改变公开主线；
- 每次发布形成不可变 snapshot，展示新增、修订、合并、拆分和撤回 diff；
- 公开页支持按主线、阶段、时间、Actor、地区和受众筛选；
- 历史数据按月份/年份静态分片，首屏不加载两年全量；
- 事件 URL 稳定；合并/撤回保留 redirect/tombstone 和修订说明。

## 13. 验收标准

### 覆盖

- 2024-07 至 2026-07 的 25 个日历月全部有 coverage record；
- 每个主线-季度单元必须是 `covered`、`sparse` 或 `evidence_gap`，不能静默空白；
- 中国与全球来源覆盖分别统计，Actor catalog 不等于有效观测；
- 每个季度至少完成一次人工 coverage review。

### 事实与回源

- 公开历史 Event 100% 满足事实发布门槛；
- 100% 有时间精度、`asOf`、原始证据状态和最后核验时间；
- 聚合站 primary evidence count 恒为 0；
- 重大数字、融资、监管和能力 claim 可追到原始文件或明确缺档说明。

### 去重与聚类

- golden set 至少覆盖 300 组同事件/不同事件/跨语种/版本升级样本；
- 文档 URL/content 去重 precision >= 0.99；
- Event pairwise F1 >= 0.90；
- 严重 false merge <= 0.5%，且所有人工修正可回放。

### 主线与总结

- 每个公开 Event 至少进入一条主线；跨线关系不复制事实；
- 10 条主线均具有季度 Phase，总结引用 3–7 个审核节点或明确证据不足；
- 当时视角与回看验证在 UI/DTO 中可区分；
- 中国追赶阶段全部绑定能力/产品/价格/商业/供应链证据。

### 持续发布

- 实时采集在 backfill 压力下仍满足既定 freshness SLO；
- backfill 可 checkpoint、暂停、重试和幂等恢复；
- snapshot 可预览、diff、批准、发布和回滚；
- 同一输入与策略版本产生相同公开 snapshot hash；
- 静态导出不包含 raw payload、内部错误、token、本机路径或未审核内容。

## 14. 成功定义

两年回溯完成不等于“库里有很多事件”。完成意味着用户可以沿任一主线回答：

1. 2024-07 时行业处于什么状态？
2. 两年间发生了哪些可验证转折？
3. 技术、资本、产品和商业化如何相互传导？
4. 中国在哪些维度跟随、收窄、并跑、领先或受限？
5. 哪些判断后来得到验证，哪些被反证？
6. 2026-07 之后最值得继续观察什么？
